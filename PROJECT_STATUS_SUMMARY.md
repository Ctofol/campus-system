# Campus Sports Health MVP - Project Status Summary

**Date:** 2026-02-14
**Version:** 1.0.0 (Pre-Phase 4)

本文档旨在对项目当前的**已实现功能**进行全面盘点，按照用户角色和技术维度分类，并明确标注开发状态。

---

## 1. 👨‍🎓 学生端 (Student Side)

**核心定位**: 运动数据记录与任务执行。

| 模块 | 功能点 | 状态 | 详细描述 |
| :--- | :--- | :--- | :--- |
| **运动类** | **户外跑步** | 🟢 **完全开发** | - 支持 GPS 轨迹记录、实时配速/距离统计。<br>- 支持定点打卡 (Checkpoints) 判定。<br>- 运动结束后数据自动上传云端。 |
| | **体测/训练** | 🟡 **部分打通** | - 仅有手动提交入口。<br>- **痛点**: 缺乏 AI 识别，目前依赖学生自觉或手动输入数据。 |
| | **历史记录** | 🟢 **完全开发** | - 完整展示历史运动列表、单次运动详情（轨迹地图）。 |
| **任务类** | **我的任务** | 🟢 **完全开发** | - 自动拉取教师发布的任务。<br>- 智能判定任务状态（进行中/已完成/过期）。 |
| **互动类** | **健康报备** | 🟢 **完全开发** | - 请假/伤病申请流程闭环（提交 -> 教师审批 -> 状态更新）。 |
| **个人中心** | **基础信息** | 🟢 **完全开发** | - 登录/注册（手机号+验证码）、头像/昵称展示。 |

---

## 2. 👩‍🏫 教师端 (Teacher Side)

**核心定位**: 教学管理与数据监控。

| 模块 | 功能点 | 状态 | 详细描述 |
| :--- | :--- | :--- | :--- |
| **任务管理** | **发布任务** | 🟢 **完全开发** | - 支持按班级或全员发布。<br>- 可设定目标（距离/时长/次数）和截止时间。 |
| | **任务统计** | 🟢 **完全开发** | - 实时查看任务完成率、参与人数。 |
| **审批管理** | **活动审批** | 🟢 **完全开发** | - 审核学生提交的运动记录（通过/驳回）。 |
| | **请假审批** | 🟢 **完全开发** | - 审核学生的健康报备申请。 |
| **学员管理** | **数据查看** | 🟢 **完全开发** | - 查看班级列表、学生个人档案。<br>- 支持按“健康状态”（正常/伤病）筛选学生。 |
| | **批量操作** | 🟢 **完全开发** | - 支持批量修改学生状态、导出学生数据 (CSV)。 |
| **测试监控** | **实时大屏** | ⚪ **仅 Demo** | - **已决定移除**: 目前为纯前端模拟数据，无真实后端对接，计划在 Phase 4 中精简掉。 |

## 3. 🛠️ 技术层面 (Technical Status)

| 维度 | 现状 | 备注/痛点 |
| :--- | :--- | :--- |
| **终端适配** | **App (Android)** | ✅ 核心适配对象，功能最全。 |
| | **H5** | ✅ 已适配，但无 GPS 定位能力（受限于浏览器）。 |
| | **小程序** | ⏳ 暂未专项适配，代码结构支持但未测试。 |
| **基础接口** | **GPS 定位** | ✅ 已对接 `uni.getLocation` / `uni.onLocationChange`。 |
| | **地图服务** | ✅ 已集成高德地图 SDK（轨迹绘制）。 |
| | **文件存储** | ✅ 已实现本地文件上传 (FastAPI Static Files)。 |
| **技术痛点** | **AI 识别** | ❌ **缺失**: 尚无端侧或云侧 AI 算法，无法自动识别深蹲/跳绳动作。 |
| | **实时性** | ❌ **弱**: 缺乏 WebSocket，消息通知依赖轮询或手动刷新。 |

## 4. 🛑 未落地 / 仅规划 (Pending / Backlog)

以下功能曾出现在早期规划中，但目前状态为**未开发**或**搁置**：

1.  **AI 智能警务训练 (`ai-police`)**:
    *   *状态*: **已搁置**。
    *   *原因*: 聚焦通用高校场景，不再做特定警务适配。
2.  **实时测试监控**:
    *   *状态*: **计划删除**。
    *   *原因*: 需求伪需求，且实现成本高（需推流），Phase 4 将移除。
3.  **课程/视频教学**:
    *   *状态*: **规划中 (Phase 4)**。
    *   *计划*: 将作为 Phase 4 的核心增量功能进行开发。
4.  **社交圈/动态**:
    *   *状态*: **低优先级**。
    *   *原因*: 优先保证教学工具属性，社交属性暂不作为 MVP 重点。

---

## 5. 总结 (Conclusion)

目前的系统是一个**“工具属性强，教学闭环完整”**的 MVP 版本。
*   **强项**: 跑步记录、任务下发、审批流。
*   **弱项**: 缺乏内容（课程）、缺乏智能化（AI）、主页引导性不强。

**Phase 4 的核心使命**:
1.  **UI/UX**: 改造主页，强化“行动导向”。
2.  **Feature**: 补齐“课程学习”板块，完善“学练结合”闭环。
3.  **Cleanup**: 清理冗余（如监控大屏），聚焦通用高校场景。
