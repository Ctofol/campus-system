# 学生端内测流程与反馈文档

版本：v1.1  
适用阶段：学生端内测  
适用对象：测试负责人、项目负责人、学生端内测参与者

## 1. 测试目标

本文档说明本次学生端内测怎么组织、重点测什么、问题怎么反馈。测试目标是验证学生端安装包在真实使用场景下是否可安装、可登录、可完成核心业务，并及时收集影响使用的问题。

本次测试重点不是教师端或管理端。教师端由项目负责人操作，用于提前发布任务、审核报备、查看学生提交结果。管理端不纳入本次测试范围。

## 2. 测试流程总览

本次测试按以下流程执行：

1. 负责人准备安装包、测试账号、测试任务、课程、跑团和反馈渠道。
2. 参与者安装 App，并确认可以正常打开。
3. 参与者使用学生账号登录。
4. 参与者按操作文档完成学生端主要功能体验。
5. 参与者重点测试登录、跑步、体测、任务、历史记录、健康报备等核心流程。
6. 发现问题后按反馈模板提交截图、录屏和复现步骤。
7. 负责人每日汇总问题，标记严重级别。
8. 开发修复后，负责人安排回归测试。

## 3. 测试范围

| 模块 | 测试内容 |
| --- | --- |
| 登录退出 | 学生账号登录、错误提示、退出登录、登录状态保持 |
| 首页 | 数据概览、快捷入口、任务和通知入口 |
| 跑步 | 自由跑、校园打卡、定位、轨迹、计时、配速、提交、结算 |
| AI 体测 | 项目选择、相机或相册授权、视频录制或上传、分析状态 |
| 我的任务 | 任务列表、任务详情、去完成、完成后状态更新 |
| 历史记录 | 运动记录、体测记录、任务记录、详情查看 |
| 课程 | 课程列表、课程详情、加入课程、视频播放 |
| 跑团 | 跑团列表、加入退出、活动报名、排行榜 |
| 健康报备 | 填写申请、上传附件、提交、查看审批状态 |
| 个人资料 | 头像、昵称、签名、资料保存 |
| 兼容与体验 | 页面加载、弱网、权限拒绝、白屏、卡顿、样式错位 |

不测试内容：

- 教师端页面操作。
- 管理端页面操作。
- 后台接口专项压测。
- 数据库或服务器运维操作。

## 4. 测试重点

请优先关注以下测试重点：

1. 安装包是否能正常安装、首次启动是否闪退。
2. 学生账号是否能正常登录、退出后是否能重新登录。
3. 首页是否能正常加载，主要入口是否可点击。
4. 自由跑是否能正常定位、开始、计时、结束和提交。
5. 运动提交后，历史记录是否能看到本次记录。
6. AI 体测是否能打开相机或相册，视频是否能上传。
7. 我的任务是否能看到负责人发布的任务，完成后状态是否更新。
8. 健康报备是否能填写、上传图片和提交。
9. 课程、跑团等列表是否能打开，详情是否能正常展示。
10. App 是否出现白屏、卡死、闪退、页面错位、按钮点不到、提示不清楚等体验问题。

## 7. 测试准备

### 7.1 项目负责人准备

1. 确认学生端访问地址或安装方式可用。
2. 确认安装包版本号、发布时间和下载方式。
3. 确认 Android 安装包可安装，iOS 安装包或 TestFlight 可正常打开。
4. 确认学生账号可登录。
5. 提前发布至少 1 个跑步任务。
6. 提前发布至少 1 个体测任务。
7. 确认课程列表中有可查看的课程。
8. 确认跑团列表中有可加入的跑团。
9. 准备问题反馈表或收集群。
10. 告知参与者测试时间、账号、安装方式、反馈格式和注意事项。

### 7.2 参与者准备

1. 使用负责人提供的账号登录。
2. 安装负责人提供的最新安装包。
3. 开启定位、相机、相册权限。
4. 准备一张测试图片。
5. 准备一个短视频，或在测试时现场录制。
6. 测试过程中尽量截图或录屏。

## 8. 测试轮次

### 8.1 第一轮：快速冒烟

目标：确认学生端核心流程能跑通。

建议耗时：30 到 60 分钟。

执行内容：

1. 登录学生端。
2. 查看首页。
3. 完成一次自由跑并提交。
4. 查看历史记录。
5. 查看我的任务。
6. 打开课程列表。
7. 打开跑团列表。
8. 提交一次健康报备。

通过标准：

- 学生可以登录。
- 首页可打开。
- 跑步可以提交。
- 历史记录能看到提交结果。
- 任务、课程、跑团、健康报备页面可进入。

### 8.2 第二轮：学生主流程测试

目标：覆盖学生端主要功能。

建议耗时：1 到 2 天。

执行内容：

- 按第 8 章测试清单逐项执行。
- 每位参与者至少完成一次跑步记录。
- 至少 2 名参与者完成体测上传。
- 至少 2 名参与者完成任务流程。
- 至少 1 名参与者提交健康报备并由负责人审核。

通过标准：

- P0 问题为 0。
- P1 问题已修复或有明确规避方式。
- 大部分参与者能独立完成学生端核心流程。

### 8.3 第三轮：真实场景测试

目标：验证真机定位、上传和网络体验。

建议耗时：1 天。

执行内容：

1. 室外自由跑测试。
2. 校园打卡点附近测试，如环境支持。
3. 4G/5G 网络下上传体测视频。
4. 弱网或网络切换时提交运动记录。
5. 负责人在教师端查看学生提交结果。

通过标准：

- 定位和轨迹基本可用。
- 视频上传有明确状态。
- 网络异常时不应白屏或数据静默丢失。
- 负责人能在教师端看到学生提交数据。

### 8.4 第四轮：修复后回归

目标：验证已修复问题。

建议耗时：0.5 天。

执行内容：

1. 复测所有 P0、P1 问题。
2. 重新执行登录、跑步提交、任务查看、历史记录。
3. 抽测体测上传、课程、跑团、健康报备。

通过标准：

- 已修复问题不再复现。
- 核心流程没有新增阻塞问题。

## 9. 问题严重级别

| 级别 | 定义 | 示例 | 处理建议 |
| --- | --- | --- | --- |
| P0 阻塞 | 学生端核心流程完全不可用 | 无法登录、跑步无法提交、首页白屏 | 立即修复 |
| P1 严重 | 核心功能异常，但可临时绕过 | 任务状态不更新、视频上传失败、历史记录缺失 | 优先修复 |
| P2 一般 | 普通功能缺陷或明显体验问题 | 文案错误、按钮位置不合理、提示不清楚 | 排期修复 |
| P3 建议 | 优化建议 | 页面更美观、增加说明、交互优化 | 视情况处理 |

## 11. 测试反馈要求

反馈问题时请尽量做到：

1. 一个问题单独反馈一次，不要把多个问题混在一起。
2. 必须说明手机型号、系统版本、安装包版本和测试账号。
3. 必须写清楚问题发生在哪个页面。
4. 必须写清楚复现步骤，最好从打开 App 或进入页面开始描述。
5. 尽量提供截图或录屏，跑步、上传、闪退类问题优先提供录屏。
6. 如果问题不是每次都出现，请说明大概出现概率。
7. 如果问题发生在跑步或定位场景，请说明室内/室外、网络环境和定位权限状态。
8. 如果问题发生在视频上传场景，请说明视频大小、网络环境和等待时间。

## 12. 执行流程

### 12.1 测试开始前

1. 项目负责人确认环境可用。
2. 项目负责人发布测试任务。
3. 项目负责人发放账号和入口。
4. 参与者确认设备权限。
5. 参与者阅读操作文档。

### 12.2 测试执行中

1. 参与者按操作文档逐项体验。
2. 发现问题立即截图或录屏。
3. 按反馈格式提交问题。
4. 项目负责人每日汇总问题。
5. 对 P0、P1 问题及时通知开发。

### 12.3 测试结束后

1. 项目负责人整理问题列表。
2. 标记问题严重级别。
3. 开发修复后安排回归。
4. 邀请部分参与者复测重点问题。
5. 输出内测结论。

## 15. 学生端内测通过标准

达到以下条件时，可认为本轮学生端内测通过：

1. 学生账号可以正常登录和退出。
2. 首页、任务、课程、跑团、历史记录等主要页面可以打开。
3. 自由跑可以开始、结束、提交，并能在历史记录查看。
4. 体测视频可以上传，上传后有明确状态。
5. 教师发布的任务能在学生端看到，并能完成至少一种任务。
6. 健康报备可以提交，审批后状态能同步。
7. P0 问题为 0。
8. P1 问题已修复，或有明确规避方案。
